# Team Orchestrator Reference

## 完整工作流程详解

### 软件开发全生命周期

#### 阶段 1: 需求分析与 PRD (10-15%)

**目标**: 明确做什么、为什么做、做成什么样

**参与角色**:
- **产品经理** (主导): 100% 投入
- **技术负责人** (参与): 20-30% 投入，评估可行性
- **设计师** (可选): 提前了解项目

**关键活动**:
1. 用户调研和需求收集
2. 竞品分析
3. 需求优先级排序 (MoSCoW 法则)
4. 编写 PRD 文档
5. 设计产品原型
6. 需求评审会议

**产出物清单**:
- [ ] PRD 文档（含背景、目标、功能列表、详细需求）
- [ ] 产品原型（Axure/Figma/墨刀）
- [ ] 用户故事地图
- [ ] 业务流程图
- [ ] 数据埋点方案
- [ ] 验收标准

**PRD 质量检查清单**:
- [ ] 背景和目标清晰
- [ ] 用户群体明确
- [ ] 功能优先级标注
- [ ] 每个功能有验收标准
- [ ] 异常流程有说明
- [ ] 非功能需求（性能、安全）有定义

---

#### 阶段 2: UI/UX 设计 (15-20%)

**目标**: 将需求转化为视觉和交互设计

**参与角色**:
- **UI/UX 设计师** (主导): 100% 投入
- **产品经理** (参与): 30% 投入，确认设计方向
- **前端负责人** (参与): 10% 投入，评估实现可行性

**关键活动**:
1. 设计需求沟通
2. 情绪板和风格探索
3. 低保真原型
4. 高保真设计
5. 交互原型制作
6. 设计评审和修改
7. 设计交付

**产出物清单**:
- [ ] 视觉设计规范（色彩、字体、间距）
- [ ] 完整页面设计稿
- [ ] 交互原型（可点击演示）
- [ ] 图标和切图资源
- [ ] 动效说明文档
- [ ] 设计系统组件库（如有）

**设计交付检查清单**:
- [ ] 所有状态已设计（正常、加载、空、错误）
- [ ] 响应式断点已标注
- [ ] 切图资源已导出并命名规范
- [ ] 特殊效果已说明实现方式
- [ ] 前端开发已确认理解设计

---

#### 阶段 3: 开发实现 (40-50%)

**目标**: 将设计转化为可运行的软件

**参与角色**:
- **前端工程师**: 100% 投入
- **后端工程师**: 100% 投入
- **产品经理**: 30-50% 投入，需求澄清和验收
- **测试工程师**: 50% 投入（中期介入）
- **设计师**: 10% 投入，UI 走查

**关键活动**:
1. 技术方案设计和评审
2. 数据库设计
3. API 接口定义
4. 前后端并行开发
5. 接口联调
6. Code Review
7. 单元测试编写

**产出物清单**:
- [ ] 前端代码（含单元测试）
- [ ] 后端代码（含单元测试）
- [ ] 数据库脚本
- [ ] API 文档（Swagger/YApi）
- [ ] 技术方案文档
- [ ] Code Review 记录

**开发完成检查清单 (DoD)**:
- [ ] 所有功能已实现
- [ ] 单元测试通过，覆盖率>70%
- [ ] Code Review 完成并合并
- [ ] 无 P0/P1级别 Bug
- [ ] 性能指标达标
- [ ] 文档已更新

---

#### 阶段 4: 测试与修复 (20-25%)

**目标**: 发现并修复缺陷，确保质量

**参与角色**:
- **测试工程师** (主导): 100% 投入
- **前端工程师**: 50% 投入，Bug 修复
- **后端工程师**: 50% 投入，Bug 修复
- **产品经理**: 30% 投入，验收测试

**关键活动**:
1. 编写测试计划和用例
2. 执行功能测试
3. 执行集成测试
4. 性能测试
5. 安全测试
6. 兼容性测试
7. Bug 修复和回归

**产出物清单**:
- [ ] 测试计划
- [ ] 测试用例
- [ ] 测试报告（功能、性能、安全）
- [ ] Bug 列表和跟踪记录
- [ ] 回归测试报告
- [ ] 上线建议书

**测试覆盖检查清单**:
- [ ] 正常流程测试通过
- [ ] 异常流程测试通过
- [ ] 边界值测试通过
- [ ] 兼容性测试通过（浏览器、设备）
- [ ] 性能测试达标（响应时间、并发）
- [ ] 安全测试无高危漏洞

---

#### 阶段 5: 部署与上线 (5-10%)

**目标**: 将软件部署到生产环境，稳定运行

**参与角色**:
- **运维工程师** (主导): 100% 投入
- **后端工程师**: 30-50% 投入，技术支持
- **测试工程师**: 30% 投入，线上验证
- **产品经理**: 10% 投入，功能确认

**关键活动**:
1. 生产环境准备
2. CI/CD流水线配置
3. 数据迁移（如需要）
4. 灰度发布
5. 全量上线
6. 监控告警配置
7. 线上验证

**产出物清单**:
- [ ] 生产环境
- [ ] CI/CD流水线
- [ ] 监控大盘和告警规则
- [ ] 运维手册
- [ ] 用户使用文档
- [ ] 上线报告

**上线检查清单**:
- [ ] 所有 P0/P1 Bug 已修复
- [ ] 性能测试达标
- [ ] 监控告警已配置
- [ ] 回滚方案已准备
- [ ] 相关人员已通知
- [ ] 线上验证通过

## 团队规模估算模型

### 基于功能点的估算法

**公式**: 
```
总工时 = Σ(功能点 × 复杂度系数 × 角色系数)
```

**功能点分类**:
| 类型 | 描述 | 基准工时 |
|-----|------|---------|
| 简单页面 | 静态展示，少量交互 | 4-8h |
| 表单模块 | 增删改查，基础验证 | 16-24h |
| 列表模块 | 分页、筛选、排序 | 16-32h |
| 复杂交互 | 拖拽、动画、实时协作 | 40-120h |
| 第三方集成 | 支付、地图、IM | 20-60h |
| 报表统计 | 数据可视化、导出 | 24-60h |

**复杂度系数**:
- 简单：0.8
- 中等：1.0
- 复杂：1.5
- 极复杂：2.5

**角色系数**:
- 产品经理：0.2-0.3（相对于开发）
- 设计师：0.3-0.5
- 前端开发：1.0
- 后端开发：1.0-1.2
- 测试：0.3-0.5
- 运维：0.2-0.3

## 约束条件处理策略

### 预算不足

**识别**: `budget < estimated_cost * 0.7`

**应对策略**:
1. 缩减范围：砍掉低优先级功能
2. 延长工期：减少并行，降低人力成本
3. 分阶段实施：先做核心功能（MVP），后续迭代
4. 使用成熟方案：减少自研，多用 SaaS/开源

**沟通话术**:
> "根据您的需求，标准配置需要 X 万元。考虑到预算限制，我们建议：
> 1. 优先保证核心功能 A、B、C
> 2. 功能 D、E 可以放到二期
> 3. 这样可以将成本控制在 Y 万元，同时保证核心价值"

### 时间紧急

**识别**: `deadline < standard_duration * 0.7` OR `urgent=true`

**应对策略**:
1. 增加人力：1.5x 配置，并行工作
2. 简化流程：采用 Rapid Response 模式
3. 削减文档：最小化文档，专注开发
4. 每日交付：每晚同步进度，及时调整

**风险提示**:
- 加急费率：成本增加 30-50%
- 质量风险：测试时间压缩，Bug 可能遗漏
- 技术债务：为速度妥协代码质量，需后续重构

### 质量要求高

**识别**: `quality_requirement = high` OR 行业强监管

**应对策略**:
1. 强制配置专职测试工程师
2. 增加代码审查轮次
3. 提高测试覆盖率要求（>80%）
4. 引入第三方安全审计
5. 延长测试周期

**额外成本**:
- 测试人力：+50-100%
- 安全审计：+10-20%
- 性能优化：+15-25%

## 协作模式对比

| 维度 | Agile Sprint | Rapid Response | Waterfall |
|-----|-------------|----------------|-----------|
| 周期 | 2 周/Sprint | 1 天/迭代 | 一次性交付 |
| 流程开销 | 中 | 低 | 高 |
| 文档要求 | 适中 | 最小化 | 完整 |
| 变更友好度 | 高 | 中 | 低 |
| 适用场景 | 常规产品 | 紧急/MVP | 需求明确 |
| 团队规模 | 3-9 人 | 2-5 人 | 不限 |
| 客户参与度 | 高 | 极高 | 低 |

## 常见问题 FAQ

### Q1: 如何确定 MVP 的范围？

**A**: 遵循以下原则：
1. **单一核心价值**: 只解决一个核心问题
2. **最小功能集**: 没有这个功能产品就无法运转
3. **可验证假设**: 能快速验证商业假设
4. **2-4 周可交付**: 超过 4 周就不算 MVP 了

**示例**:
- 电商 MVP: 商品展示 + 购物车 + 支付（不需要：优惠券、积分、秒杀）
- 社交 MVP: 发布内容 + 关注 + 点赞评论（不需要：直播、打赏、商城）

### Q2: 什么时候需要配置专职测试？

**A**: 以下情况建议配置专职 QA：
- 项目复杂度达到 medium 及以上
- 涉及金钱交易或敏感数据
- 多模块集成，测试场景复杂
- 工期>4 周，有足够测试时间
- 质量要求高（企业级、金融、医疗）

### Q3: 运维工程师何时介入？

**A**: 
- **MVP 项目**: 上线前 3-5 天介入即可
- **完整应用**: 开发中期介入（Sprint 2-3）
- **大型系统**: 需求阶段就应参与架构讨论

### Q4: 如何处理需求变更？

**A**: 
1. **评估影响**: 对进度、成本、质量的影响
2. **优先级排序**: 与 Product Owner 确认是否必须本期做
3. **正式流程**: 变更记录、同步全员、调整计划
4. **控制频率**: 一个 Sprint 内不超过 2 次重大变更

### Q5: 代码审查怎么做？

**A**: 
**基本流程**:
1. 开发者提交 Merge Request
2. 自动触发 CI（构建 + 测试）
3. 至少 1 名同事 Review（复杂功能 2 人）
4. Review 意见在 24 小时内回复
5. 修改后重新提交
6. Review 通过后合并

**Review 要点**:
- 代码正确性（逻辑是否正确）
- 代码规范（命名、格式、注释）
- 可测试性（是否有单元测试）
- 性能考虑（有无明显性能问题）
- 安全性（有无安全隐患）
