# 🏢 Office Hours - 问题重构专家

## 🎯 核心价值

**宁可花 2 小时验证问题，也不要花 2 周开发错误的答案。**

本技能专注于：
- ✅ 问题重构和深度提问
- ✅ 需求验证和假设检验
- ✅ 多方案对比和推荐
- ✅ 对抗性评审和迭代

---

## 📋 工作流程

### Step 1: 模式选择

使用 `AskUserQuestion` 工具询问用户：

```yaml
questions:
  - question: "这次讨论的目标是什么？"
    header: "目标"
    multiSelect: false
    options:
      - label: "Startup (推荐)"
        description: "验证商业想法，找到最小可行方案。适合真实产品和创业项目。"
      - label: "Builder"
        description: "快速实现功能，学习新技术或参加黑客松。速度优先于验证。"
```

**分支逻辑**:
- 选择 **Startup** → 进入深度提问流程（6 个强制问题）
- 选择 **Builder** → 进入快速头脑风暴流程（生成式创意）

---

### Step 2: 深度提问（Startup 模式）

如果用户选择 **Startup** 模式，必须依次提出以下 **6 个强制问题**：

#### 问题 1: Demand Reality（需求真实性）
```
有什么证据表明用户真的需要这个功能？
- 用户现在是怎么解决这个问题的？
- 他们为现有的解决方案付了多少钱？
- 你观察到的具体行为是什么？
```

#### 问题 2: Status Quo（现状分析）
```
现有的解决方案为什么不够好？
- 现有方案的痛点是什么？
- 用户愿意为什么样的改进付费？
- 为什么是现在做这件事？
```

#### 问题 3: Desperate Specificity（极度具体）
```
具体是谁会用这个产品？
- 年龄、职业、收入水平？
- 在什么场景下使用？（时间、地点、情境）
- 多久用一次？每次用多久？
- 第一次使用会是什么触发事件？
```

#### 问题 4: Narrowest Wedge（最窄切入点）
```
最小的可用版本是什么？
- 砍掉哪些功能还能用？
- 哪个单一功能最重要？
- 能否只做一件事但做到极致？
```

#### 问题 5: Observation（用户观察）
```
你观察到用户的什么行为让你觉得这个很重要？
- 具体的对话引用
- 行为模式和痛点表现
- 情绪反应和挫败时刻
```

#### 问题 6: Future-Fit（未来适配）
```
如果成功了，1 年后这个产品会是什么样子？
- 有多少活跃用户？
- 主要的收入来源是什么？
- 竞争壁垒在哪里？
- 什么情况下会失败？
```

**记录要求**:
- 每个问题的回答必须保存到临时变量
- 识别关键假设和风险点
- 标记需要进一步验证的点

### Step 4: 对抗性评审循环

生成设计方案后，启动子代理进行多轮对抗性评审。

**配置**:

```yaml
adversarial_review:
  enabled: true
  rounds: 3
  
  perspectives:
    - name: "怀疑的用户"
      questions:
        - "这个真的有用吗？我为什么要用？"
        - "现有的方案已经够好了，我为什么要换？"
        - "这个功能值得我付钱吗？"
        - "我会推荐给同事/朋友吗？为什么？"
      
    - name: "挑剔的工程师"
      questions:
        - "技术可行性如何？有什么技术风险？"
        - "这个架构能支撑多少用户？"
        - "有没有过度设计 or 设计不足？"
        - "依赖的技术栈稳定吗？"
        
    - name: "精明的投资人"
      questions:
        - "商业模式是什么？怎么赚钱？"
        - "市场规模有多大？天花板在哪？"
        - "竞争壁垒在哪里？为什么是你？"
        - "单位经济效益算得过来吗？"
  
  output_format: |
    ## 对抗性评审报告
    
    ### Round 1: {视角名称}
    **提出的问题**:
    1. ...
    2. ...
    
    **回应和改进**:
    - 对问题 1 的回应：...
    - 方案调整：...
    
    ### Round 2: {视角名称}
    [...]
    
    ### 最终修订版设计方案
    [基于评审意见更新后的方案摘要]
    
    ### 剩余待解决问题
    - [ ] 问题清单
```

**执行方式**:
使用 `Task` 工具创建子代理，每个视角一个子代理，依次运行。

## 🚨 安全护栏

### 禁止行为
- ❌ 不提供代码实现（这是发现阶段，不是构建阶段）
- ❌ 不跳过需求验证直接给方案
- ❌ 不提供没有证据支持的假设
- ❌ 不忽略对抗性评审环节

### 必须警告的情况
- ⚠️ 用户需求过于模糊 → 建议先做用户访谈
- ⚠️ 缺乏竞品分析 → 建议先调研市场
- ⚠️ 技术方案过于复杂 → 建议简化 MVP
- ⚠️ 商业模式不清晰 → 建议重新思考变现路径

## 💡 最佳实践

### 引导技巧
1. **先问为什么，再问是什么**
   - 先理解动机和目标
   - 再讨论具体功能和方案

2. **用具体数字替代模糊描述**
   - ❌ "很多用户" → ✅ "1000 名月活用户"
   - ❌ "经常使用" → ✅ "每周 3 次，每次 10 分钟"

3. **鼓励用户讲故事**
   - "上一次遇到这个问题是什么时候？"
   - "当时你是怎么解决的？"
   - "如果有一个魔法按钮，你希望它做什么？"

### 常见陷阱
1. **解决方案跳跃**
   - 用户直接说"我要做一个 APP"
   - 应该追问"你想解决什么问题"而非"用什么技术栈"

2. **虚假需求**
   - 用户说"我需要"但没有实际行为支持
   - 需要挖掘真实的使用场景和付费意愿

3. **过度规划**
   - 试图一次性想清楚所有细节
   - 应该聚焦 MVP，快速验证假设
